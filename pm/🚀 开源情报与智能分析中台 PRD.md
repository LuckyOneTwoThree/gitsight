## 第一章：项目背景与产品目标

### 1.1 背景与立项原因

GitHub 拥有超过 3 亿个代码仓库，每天有大量高价值的开源项目涌现。然而，信息过载已成为开发者、产品经理、技术投资人最核心的痛点：

- **发现难**：GitHub 官方 Trending 只按 Star 排序，无法过滤刷榜项目，无法按语义搜索，无法追踪短期飙升的黑马项目。
- **理解难**：一个几十万行代码的大型仓库，即使是资深工程师也需要数天乃至数周才能基本搞懂架构意图。
- **决策难**：产品经理做竞品调研，投资人判断技术赛道，都需要花费大量时间手动整合多个维度的信息，缺乏结构化输出。

本产品旨在利用 AI 能力，将 GitHub 仓库从「非结构化代码库」转化为「结构化商业与技术洞察」，消除开源世界的信息差与理解门槛。

### 1.2 产品定位

> **一句话定位**：开源桌面端 AI 情报分析工具——专注全球开源项目的深度解析与赛道洞察，全程 AI 驱动，数据本地存储。

**产品愿景**：消除开源世界的"信息差"与"理解门槛"，将非结构化的代码仓库转化为结构化的商业与技术洞察。

本产品是一个面向开发者、产品经理和技术投资人的**开源桌面端** GitHub 项目智能分析工具，覆盖「发现热门项目 → 读懂单个项目 → 横向对比决策 → 洞察整个赛道」的全链路。采用 Electron 桌面端架构，所有数据存储在本地 SQLite 数据库，用户自带 LLM API Key，无需注册登录即可使用。

**核心特性**：
- 支持 15+ LLM 提供商（OpenAI、Anthropic、Google、DeepSeek、Qwen、Zhipu、Kimi、MiMo、Volcengine、Baichuan、Yi、StepFun、MiniMax、SiliconFlow、OpenRouter、Custom）
- Electron 桌面应用，本地 SQLite 存储，数据完全由用户掌控
- 中英文双语界面，报告输出语言严格跟随用户选择
- Fast/Deep 双模式分析，按需生成 + 全量缓存

## 第二章：产品功能架构总览

整个系统按照用户使用漏斗，划分为四大核心模块，形成完整的情报闭环。**当前 V1.0 版本四个模块均已实现**：

|模块编号|模块名称|核心价值|主要用户|实现状态|
| ----------| ----------------------------------------------| --------------------------------| ----------------------| --------|
|M1|项目雷达与大盘（Discovery）|发现热门、新兴、黑马开源项目|所有用户|已实现|
|M2|单点深度解析（Deep Dive）|将单个 Repo 转化为多维分析文档|PM / 开发者 / 投资人|已实现|
|M3|横向对比与机会挖掘（Comparison）|多项目竞品分析 + 机会输出|PM / 创业者|已实现|
|M4|赛道全局视图与情报订阅（Landscape & Alerts）|行业全局视野 + 个性化情报推送|投资人 / 行业分析师|已实现|

**模块间关系**：Discovery 是用户入口，Deep Dive 是核心价值交付，Comparison 是商业价值放大，Landscape & Alerts 是留存与复访驱动。四个模块形成「发现 → 解析 → 决策 → 持续关注」的闭环。

**跨模块功能**：
- **收藏系统（Watchlist）**：一键收藏感兴趣的项目，支持 GitHub Star 同步（配置 GitHub Token 后，本地收藏自动同步到 GitHub Star）
- **Fast/Deep 分析模式**：快速模式生成精简报告，深度模式生成完整报告
- **中英文语言切换**：报告输出语言严格跟随用户选择
- **数据备份与恢复**：支持全量数据备份和恢复，包含项目数据、分析报告、收藏和订阅规则

### 3.2 模块 M2：单点深度解析（Deep Dive）

#### 3.2.1 功能概述

M2 是产品的核心价值交付模块。用户进入某个 Repo 的详情页后，可按需触发不同视角的 AI 深度分析，生成结构化文档。采用「按需生成 + 全量缓存」策略降低 AI Token 成本。支持 Fast（快速分析）和 Deep（深度分析）两种模式，支持中英文报告输出。

> **核心设计原则**：一键生成千人千面的多维分析包——不同角色用户（PM / 开发者 / 投资人）看到的分析维度不同，可按需解锁所需视角的深度报告。

#### 3.2.2 分析视角详细说明

当前共 9 个分析视角：

|视角名称|API section_type|产出物内容|AI 数据来源|目标用户|实现状态|
| -------------------------| -----------------|---------------------------------------------------------------------------------------------| --------------------------------------------------------| -----------------| ------------|
|TL;DR 快速摘要|tldr|项目是什么、解决什么痛点、核心优势、适用场景（限 200 字）|README + 仓库描述|所有人|已实现|
|产品视角：逆向 PRD|reverse_prd|产品定位声明、功能列表（Feature List）、核心用户故事（User Story）、商业模式推断、战略判断、机会评分卡|README + Issues + Discussions + Release Notes|PM / 创业者|已实现|
|技术视角：核心架构梳理|architecture|技术栈全景图、模块拆解（核心模块职责说明）、数据流图（Mermaid Flowchart 实时渲染）、关键设计决策说明|README + package.json/go.mod + 一级目录 + 核心配置文件|开发者 / 架构师|已实现|
|技术视角：CodeWiki 导读|code_wiki|快速启动指南（避坑版）、核心模块入口文件注释导读、常见问题 FAQ（来自 Issues）|README + Issues（标签 bug/question）+ 核心代码文件|独立开发者|已实现|
|时光机分析|timeline|提取近 1-2 年核心 Release Notes，复盘「从 MVP 到现在如何演进」，推断产品路线决策逻辑|Release Notes + 重大 PR|PM / 架构师|已实现|
|技术栈全景|tech_stack|编程语言分布、核心依赖框架、部署方式、License 类型、依赖与供应链风险|package.json/go.mod/requirements.txt + 仓库描述|开发者|已实现|
|社区生态分析|community|贡献者集中度、Issue 解决速度、PR 合入周期、近期 Release 频次、社区健康度评分|Issues + PR 统计 + 贡献者数据|投资人 / 创业者|已实现|
|贡献指南|contribution_guide|如何参与贡献、Issue 标签体系说明、PR 流程、代码规范|CONTRIBUTING.md + Issues + PR 模板|开发者|已实现|
|供应链安全分析|supply_chain|依赖风险、许可证合规、维护者风险、安全建议|依赖清单 + License 信息 + 维护者数据|安全工程师 / 架构师|已实现|

#### 3.2.3 生成状态管理

|状态|说明|前端展示|
| ----------------| --------------------------------------------------------| -----------------------------------|
|未生成 (not_generated)|该视角报告从未被触发生成|显示「生成」按钮|
|生成中 (generating)|AI 正在后台异步处理（预计 30-120 秒）|显示进度条 + 分析阶段说明文字（异步轮询状态）|
|已缓存 (cached)|已生成过，直接读取缓存|立即展示，右上角标注缓存时间|
|生成失败 (failed)|AI 解析异常或 Repo 数据不足|提示失败原因，支持重试|

#### 3.2.4 Fast/Deep 分析模式

|模式|说明|适用场景|
| ----------------| --------------------------------------------------------| -----------------------------------|
|Fast（快速分析）|生成精简版报告，跳过深层分析章节，响应更快|快速了解项目概况|
|Deep（深度分析）|生成完整版报告，包含所有分析章节和深度推断|正式调研、技术选型|

#### 3.2.5 语言强制执行

报告输出语言严格跟随用户在设置中选择的界面语言（中文 / 英文）。Fast 模式下额外追加语言提醒，确保模型输出与用户选择一致。

#### 3.2.6 架构图 Mermaid 渲染

架构分析视角中的数据流图使用 Mermaid.js 实时渲染为可视化流程图，而非静态文本。支持查看 Mermaid 源码。HTML 内容通过 DOMPurify 进行安全过滤。

## 第五章：技术架构策略

### 5.1 整体架构概述

采用「Electron 桌面端 + Next.js 全栈 + SQLite 本地存储 + 多 LLM 提供商」的架构，核心设计原则是「按需拉取」和「渐进式 AI 解析」：

|层级|组件 / 技术选型|职责说明|
| --------------| --------------------------------------------------| --------------------------------------------------------------|
|桌面端|Electron 42 + electron-builder|桌面应用壳，内嵌 Next.js 服务，支持 Windows/macOS/Linux 打包分发|
|前端层|Next.js 16 + React 19 + TypeScript 5.7 + TailwindCSS 4 + Turbopack|SSR + CSR 混合渲染，shadcn/ui 组件库|
|API 服务层|Next.js API Routes|全栈 API，包含 6 个业务模块（项目/分析/对比/赛道/订阅/用户工作台）|
|AI 分析引擎|Node.js + 文件存储任务队列|异步分析任务，调用 LLM API 生成报告，前端轮询状态|
|主数据库|SQLite（better-sqlite3）+ WAL 模式|存储项目元数据、分析报告缓存、对比报告、订阅规则、收藏列表|
|LLM 接入层|多提供商适配器（OpenAI 兼容 API）|支持 15+ LLM 提供商，用户自带 API Key，统一 OpenAI Chat Completions 协议|
|配置管理|JSON 文件配置（~/.repointel/config.json）+ globalThis HMR 安全缓存|桌面端配置管理，支持多 LLM 提供商配置，HMR 热更新不丢失缓存|
|图表渲染|Recharts + Mermaid.js|数据可视化（雷达图、趋势图）+ 架构流程图渲染|
|HTML 安全|DOMPurify|AI 生成内容的 HTML 安全过滤|
|外部数据源|GitHub REST API（按需拉取）|用户访问不存在的项目时实时拉取元数据并入库|

### 5.2 与 SaaS 架构的关键差异

本产品为桌面端单用户应用，与 SaaS 架构存在以下关键差异：

|差异点|SaaS 方案|桌面端方案|
| --------------| --------------------------------------------------| --------------------------------------------------------------|
|数据库|PostgreSQL + Drizzle ORM|SQLite（better-sqlite3）+ WAL 模式|
|缓存/队列|Redis + BullMQ|无分布式缓存，文件存储任务队列|
|认证|NextAuth.js + GitHub OAuth|无认证系统，单用户桌面应用|
|API 服务|Fastify 独立服务|Next.js API Routes（全栈一体）|
|部署|多节点 + CDN|Electron 打包，本地安装|
|会话管理|数据库 Session|无会话，配置文件持久化|

### 5.3 核心技术挑战与应对策略

#### 5.3.1 GitHub API 频率限制与数据同步策略

|方案|说明|
| -----------------------------| --------------------------------------------------------------------------------------------------------------------------|
|MVP 策略：按需拉取+轻量脚本同步|用户访问不存在的项目时，实时调用 GitHub REST API 拉取元数据并入库；同时定时脚本日更 GitHub Trending Top 100 + 周更各语言 Top 50|
|缓存策略|已入库项目直接查 SQLite（< 10ms），GitHub API 响应缓存 24h，404 响应缓存 1h 防穿透|
|GitHub Token 增强|用户配置 GitHub Token 后，API 频率限制从 60 次/小时提升至 5000 次/小时，同时解锁 GitHub Star 同步功能|

#### 5.3.2 AI 上下文超限与渐进式解析策略

大型仓库（如数百万行代码）无法直接喂给 LLM，采用三层渐进式解析策略：

|解析层级|输入内容|输出报告类型|Token 消耗|
| --------------------| --------------------------------------------------------------------------| --------------------------------| ----------------------|
|第一层：元数据层|README.md + 仓库描述 + Topic 标签 + 一级目录树|TL;DR 摘要、AI 标签、项目分类|低（~500 Tokens）|
|第二层：结构分析层|README + package.json/go.mod/requirements.txt + 二级目录 + Release Notes|架构梳理、逆向 PRD、时光机分析|中（~3,000 Tokens）|
|第三层：深度代码层|RAG 技术：按需检索核心代码文件（基于 AST 解析的关键函数/类）|CodeWiki、核心模块导读|高（~10,000 Tokens）|

#### 5.3.3 AI Token 成本控制

- **全量缓存**：首次触发的分析结果立即缓存，后续直接读缓存，边际成本为零
- **增量更新**：仅当检测到项目新 Release 且内容变更超过阈值时，触发重新生成
- **Tier 化模型**：TL;DR 等轻量任务使用低成本模型，深度 PRD 等任务使用高质量模型
- **Fast/Deep 模式**：快速模式生成精简报告节省 Token，深度模式生成完整报告
- **用户自带 API Key**：用户自行承担 LLM 调用成本，无平台侧成本压力

1. 用户在搜索框输入：「开源的 AI 工作流编排框架」

5. 用户阅读对比报告 → 发现新产品切入点 → 导出 Markdown 报告

#### 流程 3：开发者快速上手陌生项目（CodeWiki 场景）

1. 用户在搜索结果页发现某个项目 → 进入详情页
2. 选择 Deep 模式 → 点击「技术视角 → CodeWiki 导读」
3. 系统判断是否有缓存：无缓存 → 触发生成
4. 生成完成（约 60 秒）→ 前端轮询获取完成状态 → 展示「核心模块注释导读 + 快速启动避坑指南」
5. 用户通过 CodeWiki 在 2 小时内跑通本地环境（节省原本需要 1-2 天的时间）

#### 流程 4：投资人持续关注技术赛道

1. 用户进入赛道页面 → 选择「AI Agent」赛道
2. 查看赛道全景气泡图 + 项目排行榜 + 赛道 AI 摘要
3. 在订阅管理中创建订阅规则：「AI Agent 赛道，周增 Star > 500」
4. 配置推送渠道（飞书 / 企微 / Discord 等）
5. 系统按规则自动推送符合条件的黑马项目
